Analogia

Legos que se conectan

Cada tabla es una pieza de Lego. Las ForeignKeys son los "puntos de conexion" que permiten unir piezas.

Student FK → student_id Enrollment FK → course_id Course

Enrollment es la pieza que conecta Student y Course.

ForeignKey

ForeignKey: el puente entre tablas

from sqlalchemy import ForeignKey

class Enrollments(Base):
    __tablename__ = "enrollments"

    id = Column(Integer, primary_key=True)
    student_id = Column(
        Integer, ForeignKey("students.id"), nullable=False
    )
    course_id = Column(
        Integer, ForeignKey("courses.id"), nullable=False
    )

ForeignKey("students.id") dice: "esta columna apunta a la columna id de la tabla students".

relationship()

relationship(): navegar entre objetos

from sqlalchemy.orm import relationship

class Student(Base):
    __tablename__ = "students"

    id = Column(Integer, primary_key=True)
    name = Column(String(80), nullable=False)
    enrollments = relationship("Enrollment",
        back_populates="student")

class Enrollment(Base):
    __tablename__ = "enrollments"

    student_id = Column(Integer, ForeignKey("students.id"))
    student = relationship("Student",
        back_populates="enrollments")

relationship() crea un atributo Python para navegar entre objetos. No es una columna.

back_populates

back_populates: conexion bidireccional

En español simple back_populates conecta dos relationship() para que SQLAlchemy sepa que son el mismo vinculo visto desde ambos lados. Asi cuando agregas un enrollment a un student, el enrollment automaticamente sabe a que student pertenece.
# Student.enrollments → lista de Enrollment
# Enrollment.student → un Student

s = db.query(Student).first()
s.enrollments  # [Enrollment, Enrollment]
s.enrollments[0].student.name  # "Ana"

Lazy vs Eager

Lazy loading vs Eager loading

AspectoLazyEager (selectinload)
Cuando cargaSolo cuando accedes a la relacionEn la misma consulta principal
Consultas SQL1 + N (problema N+1)2 (principal + JOIN)
Rendimiento con pocos datosBuenoSimilar
Rendimiento con muchos datosMalo (N+1)Bueno
 
# Lazy (default)
student.enrollments  # ← aqui hace la query

# Eager
from sqlalchemy.orm import selectinload
db.query(Student).options(
    selectinload(Student.enrollments)
).all()

N+1

Problema N+1: el enemigo del rendimiento

# ❌ N+1: 1 query para students + N queries para enrollments
students = db.query(Student).all()
for s in students:
    print(s.enrollments)  # ← N queries separadas!

# ✅ Solucion con selectinload
from sqlalchemy.orm import selectinload
students = db.query(Student).options(
    selectinload(Student.enrollments)
).all()
# Solo 2 queries total
Atencion El problema N+1 aparece cuando tienes listas y accedes a relaciones. Siempre usa eager loading cuando sepas que vas a necesitar las relaciones.

class StudentResponse(BaseModel):

    name: str

class CourseWithStudents(BaseModel):
    model_config = ConfigDict(from_attributes=True)
    id: int

    students: list[StudentResponse]

@router.get("/courses/{id}")
def read_course(id: int, db: Session = Depends(get_db)):
    course = db.query(Course).options(
        selectinload(Course.enrollments)
        .selectinload(Enrollment.student)
    ).filter(Course.id == id).first()
    return course

Asi NO vs Asi SI

Manual JOIN vs relationship

❌ JOIN manual

results = db.execute(
  "SELECT s.name FROM students s "
  "JOIN enrollments e ON s.id = e.student_id "
  "WHERE e.course_id = ?", (id,)
)

SQL plano, no portatil, propenso a errores.

✅ relationship()

course = db.query(Course).options(
  selectinload(Course.enrollments)
).first()
course.enrollments  # ✅

Python puro, legible, portatil entre BD.

Troubleshooting

Errores comunes con relaciones

ErrorSolucion
IntegrityError: FK violadaEl ID referenciado no existe. Crea primero el registro padre.
DetachedInstanceError con relacionesUsa selectinload para cargar la relacion dentro de la sesion.
La relacion devuelve NoneVerifica que los datos existan y que el FK sea correcto.
N+1 queries en loopAgrega .options(selectinload(...)) a la query principal.
RecursionError en response_modelUsa modelos separados para entrada y salida, no el mismo modelo.
