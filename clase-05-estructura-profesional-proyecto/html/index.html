Estructura profesional de proyecto

Routers, schemas, services y configuracion central para proyectos que escalan.

Ing. Eduardo Pimienta

Objetivos

Al finalizar esta clase podras

  • Organizar un proyecto FastAPI en capas: routers, schemas, services, core.
  • Usar APIRouter para agrupar endpoints por recurso.
  • Separar la logica de negocio en services.
  • Centralizar configuracion con Pydantic Settings.

Problema

❌ Monolito

Estructura

Arbol de directorios

app/
  main.py           # Punto de entrada
  core/config.py    # Configuracion central
  routers/          # Endpoints HTTP
    health.py
    courses.py
  schemas/          # Modelos Pydantic
    course.py
  services/         # Logica de negocio

main.py

main.py: el recepcionista

from fastapi import FastAPI
from app.core.config import settings
from app.routers import courses, health

app = FastAPI(
    title=settings.app_name,
    version=settings.app_version,
)

app.include_router(health.router)
app.include_router(courses.router)

main.py solo crea la app y registra routers. No tiene logica de negocio.

core/config.py

Configuracion centralizada

from pydantic import BaseModel

class Settings(BaseModel):
    app_name: str = "API academica"
    app_version: str = "0.1.0"
    environment: str = "development"
    database_url: str = "sqlite:///./cursos.db"

settings = Settings()

Toda la configuracion en un solo lugar. Facil de cambiar y mantener.

routers/

Routers con APIRouter

from fastapi import APIRouter

router = APIRouter(
    prefix="/courses",
    tags=["Courses"],
)

@router.get("/")
def list_courses():
    return [{"id": 1, "name": "Web II"}]

APIRouter agrupa endpoints de un mismo recurso. prefix y tags evitan repetir codigo.

schemas/

Schemas: contratos de datos

from pydantic import BaseModel, Field

class CourseCreate(BaseModel):
    name: str = Field(min_length=3, max_length=80)
    credits: int = Field(ge=1, le=6)

class CourseResponse(BaseModel):
    id: int
    name: str
    credits: int
    active: bool

Define la forma de los datos que entran y salen de la API.

services/

Services: logica de negocio

def create_course(data: CourseCreate) -> Course:
    if course_exists(data.name):
        raise HTTPException(400, "Curso ya existe")
    course = Course(**data.dict())
    return course

Los services contienen las reglas de negocio. Los routers solo llaman a services.

En español simple El router es el mesero (toma el pedido), el service es el cocinero (prepara la comida). El mesero no cocina.

Flujo

Flujo completo de una peticion

Request Router Service Schema Response 
GET /courses  →  courses.router  →  list_courses()
                                     →  CourseResponse[]

Cada capa tiene una responsabilidad unica. Asi el proyecto puede crecer sin romperse.

Asi NO vs Asi SI

Router gordo vs Router delgado

❌ Router gordo

@router.get("/courses")
def list():
    cursos = db.query(...)
    if not cursos:
        raise HTTPException(...)
    return [c.to_dict()
            for c in cursos]

Logica de negocio en el router. Ilegible y no testeable.

✅ Router delgado

@router.get("/courses")
def list():
    return course_service.list()

Router solo llama al service. La logica esta en el service.

Troubleshooting

Errores comunes

ErrorSolucion
Router no funcionaOlvidaste app.include_router() en main.py
ImportError circularService importa schema que importa service. Divide en archivos.
Uvicorn no encuentra appEjecuta desde la carpeta del proyecto, no desde app/
Falta __init__.pyCada carpeta Python necesita __init__.py (vacio basta)
prefix duplicadoNo pongas /courses en el router y en @router.get("/courses")

Checklist

Al salir de clase

  • [ ] Cree la estructura app/core, routers, schemas, services.
  • [ ] Agregue __init__.py en cada carpeta.
  • [ ] Cree un APIRouter con prefix y tags.
  • [ ] Centralice la config en core/config.py.
  • [ ] Los routers solo llaman services, no tienen logica.
  • [ ] Los schemas definen entrada y salida.
  • [ ] Registre los routers en main.py.
  • [ ] La app corre sin errores.

Actividad

Refactorizar a estructura profesional

  1. Toma tu API de la clase 04 (un solo main.py).
  2. Crea la carpeta app/ con la estructura vista.
  3. Mueve los endpoints a routers/ con APIRouter.
  4. Crea schemas Pydantic para entrada/salida.
  5. Crea services con la logica de negocio.
  6. Centraliza la config en core/config.py.
  7. Verifica que todo funciona con uvicorn.

Cierre

Lo que debe quedar claro

  • La estructura no es burocracia: es una necesidad para proyectos reales.
  • Router llama a service. Service tiene la logica. Schema valida datos.
  • APIRouter agrupa endpoints con prefix.
  • Configuracion central evita valores hardcodeados.
