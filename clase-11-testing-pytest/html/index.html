Testing con pytest y httpx

Calidad y confianza en tu API

Ing. Eduardo Pimienta

Motivacion

¿Por que testear APIs?

Sin tests

  • Pruebas manuales en Swagger UI
  • Confias en que "mas o menos funciona"
  • Miedo a hacer cambios grandes
  • Bugs aparecen en produccion

Con tests

  • Un solo comando: pytest
  • 30 tests en 2 segundos
  • Libertad para refactorizar
  • Bugs se detectan antes

Herramientas

pytest + httpx + TestClient

pytest Busca funciones test_* y verifica con assert
httpx Cliente HTTP moderno para Python
TestClient Envueltto de FastAPI que simula peticiones HTTP sin servidor 
pip install pytest httpx pytest-cov

Concepto clave

TestClient

Simula un navegador haciendo peticiones a tu API. No necesita uvicorn corriendo.

from fastapi.testclient import TestClient
from app.main import app

client = TestClient(app)

response = client.get("/health/")
assert response.status_code == 200
assert response.json() == {"status": "ok"}
MetodoUso
client.get(url)GET request
client.post(url, json=...)POST con body JSON
client.delete(url)DELETE request
response.status_codeCodigo HTTP
response.json()Body como dict

Estructura

Organizacion de tests

app/tests/
  __init__.py       # Vacio
  conftest.py        # Fixtures compartidas
  test_health.py     # Endpoints publicos
  test_auth.py       # Registro, login, token
  test_courses.py    # CRUD de cursos
  test_errors.py     # 404, 422, 403
  test_middleware.py # X-Process-Time, CORS

Cada archivo test_*.py es descubierto automaticamente por pytest.

Fixtures

conftest.py — Ingredientes preparados

@pytest.fixture()
def client():
    app.dependency_overrides[get_db] = override_get_db
    with TestClient(app) as c:
        yield c

@pytest.fixture()
def token(client, test_user_data):
    client.post("/users/register", json=test_user_data)
    resp = client.post("/token", data={
        "username": "ana", "password": "123456"})
    return resp.json()["access_token"]

@pytest.fixture()
def auth_headers(token):
    return {"Authorization": f"Bearer {token}"}

BD en tests

Aislar la base de datos

Cada test debe empezar con BD limpia. La fixture autouse lo garantiza:

@pytest.fixture(autouse=True)
def test_db():
    Base.metadata.create_all(bind=engine)
    yield                       # Test se ejecuta aqui
    Base.metadata.drop_all(bind=engine)

SQLite en memoria: rapido, aislado, sin efectos secundarios entre tests.

Tests

Probar health

def test_health_status(client):
    response = client.get("/health/")
    assert response.status_code == 200
    assert response.json() == {"status": "ok"}

def test_health_version(client):
    response = client.get("/health/version")
    assert response.status_code == 200
    data = response.json()
    assert "name" in data
    assert "version" in data

pytest -v -k "health" → 2 tests en 0.1 segundos

Tests

Probar autenticacion

def test_register_success(client, test_user_data):
    resp = client.post("/users/register", json=test_user_data)
    assert resp.status_code == 201
    assert resp.json()["role"] == "student"

def test_login_success(client, test_user_data):
    client.post("/users/register", json=test_user_data)
    resp = client.post("/token", data={
        "username": "ana", "password": "123456"})
    assert resp.status_code == 200
    assert "access_token" in resp.json()

def test_users_me_without_token(client):
    assert client.get("/users/me").status_code == 401

Tests

Probar endpoints protegidos

def test_create_course_without_token(client):
    resp = client.post("/courses/", json={"name": "Fisica", "credits": 4})
    assert resp.status_code == 401

def test_create_course_with_token(client, auth_headers):
    resp = client.post("/courses/", json={"name": "Fisica", "credits": 4},
                       headers=auth_headers)
    assert resp.status_code == 201

La fixture auth_headers ya incluye el token. No necesitas pensar en autenticacion en cada test.

Tests

Probar roles (admin vs student)

def test_delete_as_student(client, auth_headers, created_course):
    resp = client.delete(f"/courses/{created_course['id']}",
                         headers=auth_headers)
    assert resp.status_code == 403

def test_delete_as_admin(client, admin_headers, created_course):
    resp = client.delete(f"/courses/{created_course['id']}",
                         headers=admin_headers)
    assert resp.status_code == 200

Con fixtures separadas (auth_headers / admin_headers) probamos ambos roles sin repetir codigo.

Tests

Probar errores (404, 422, 403)

def test_404(client):
    resp = client.get("/courses/9999")
    assert resp.status_code == 404
    assert "no encontrado" in resp.json()["mensaje"]

def test_422(client, auth_headers):
    resp = client.post("/courses/", json={}, headers=auth_headers)
    assert resp.status_code == 422
    assert resp.json()["error"] is True
    assert "detalles" in resp.json()

No solo pruebes que el codigo HTTP es correcto. Verifica el formato del mensaje de error.

Tests

Probar middleware

def test_x_process_time(client):
    resp = client.get("/health/")
    assert "x-process-time" in resp.headers

def test_cors_headers(client):
    resp = client.options("/courses/",
        headers={"Origin": "http://localhost:5173",
                 "Access-Control-Request-Method": "GET"})
    assert "access-control-allow-origin" in resp.headers

Los middleware afectan Toda la API. Un par de tests verifican que estan funcionando.

Calidad

Cobertura de codigo

pytest --cov=app --cov-report=term-missing
CoberturaSignificado
100%Cada linea se ejecuto al menos una vez
80-99%Bien, pero hay codigo no probado
< 50%Tu API sin tests es una caja de sorpresas

100% de cobertura NO = 0 bugs. La cobertura solo mide que se ejecuto, no que se verifico correctamente.

Actividad

Anadir tests a tu API

  1. Instala pytest, httpx y pytest-cov.
  2. Crea app/tests/conftest.py con fixtures de BD, client, token, auth_headers.
  3. Escribe test_health.py — 2 tests minimos.
  4. Escribe test_auth.py — register, login, users/me.
  5. Escribe test_courses.py — CRUD completo con roles.
  6. Ejecuta pytest -v — todos pasan.
  7. Ejecuta pytest --cov=app — cobertura minima 80%.

Cierre

Lo que debe quedar claro

  • TestClient simula peticiones HTTP sin servidor real.
  • Las fixtures en conftest.py preparan el escenario para cada test.
  • Cada test debe empezar con BD limpia (autouse fixture).
  • Prueba tanto el exito (200/201) como los errores (401, 403, 404, 422).
  • La cobertura mide que lineas se ejecutaron, no que los tests sean correctos.
  • TDD: test primero, codigo despues.
